# SOC Alerts Dashboard - Report

## Overview

I built a full-stack SOC triage app with three core workflows: a dashboard for aggregate visibility, a filterable/paginated alerts list, and a detail view for updating alert state. The goal was to make the app feel like a practical analyst tool rather than a generic CRUD demo.

## Architecture

### Frontend
- React 18 + Vite
- React Router for navigation
- Axios client scoped to `/api`
- Tailwind CSS for the visual system

### Backend
- Express.js REST API
- SQLite persistence
- JWT stored in an httpOnly cookie
- bcryptjs for password hashing

The frontend restores the session by calling `/api/auth/me` on load, so authentication is cookie-backed rather than token-handled in browser storage.

## What I Built

- Dashboard with totals, severity/category/status breakdowns, and a simple 14-day trend
- Click-through aggregates that open the alert list with filters applied
- Alerts list with severity/status/category/search/time-range filters and severity/timestamp sorting
- Alert detail view with editable status, severity, and assignee fields
- One-click dismiss actions for resolved and false-positive outcomes
- Raw event JSON visibility for investigation work
- Seeded demo user: `analyst@company.com` / `Alert123!`
- Mock dataset of about 1000 records generated by script

## Trade-offs

- SQLite keeps deployment simple and is enough for an assessment-scale dataset
- One seeded analyst account keeps the auth story clean without overbuilding roles
- The UI focuses on the core triage loop instead of adding bulk actions or realtime streaming
- Trend and aggregate views are intentionally lightweight so the app stays fast and understandable

## What I Would Do Next

- Add automated tests for API routes and the main React flows
- Add realtime alert updates through websockets or server-sent events
- Add bulk operations and saved filter presets
- Add audit logging for alert state changes
- Replace SQLite with Postgres if the app needed multi-user scale

## Notes

I intentionally kept the code path small and conventional: one backend API, one frontend client, and URL-driven filters so views can be shared and refreshed without losing context.
